There is **no `.git` directory anywhere in this tree.** This project has no
version control at all — no commit history, no branches, no way to diff what
changed or recover a previous version. For a three-person capstone where two
other people (Daniel, Willow) need to integrate with this code, this is the
single biggest structural risk in the project, independent of code quality.

## What has been validated

- **Feature mapping correctness for synthetic input.** `tests/test_suricata_reader.py`
  confirms `flow_to_features()` produces every required feature, with sane values,
  and never divides by zero — but only against hand-written sample flows, not a
  real Suricata capture.
- **Report generation contract.** `tests/test_report.py` confirms the template
  report contains the attacker IP, attack type, score, and a "Recommended actions"
  section, and degrades gracefully on a near-empty event dict.
- **The model loads and scores in this exact environment.** Confirmed by directly
  running `predict()` against both hand-written and real-dataset-derived feature
  vectors today (see below).
- **The model genuinely separates attack-shaped flows from benign ones.** Pulling
  an actual attack row out of `Friday-WorkingHours-Afternoon-PortScan.pcap_ISCX.csv`
  and feeding its 10 feature values to `predict()` returns `score=100,
  is_response_triggered=True`. This is real evidence the trained model works on
  CICIDS2017-shaped data — it was not just asserted from the training script's own
  printout.
- **Training metrics (99.55% accuracy, 0.41% FPR)** are stated in `PROGRESS.md` and
  `TRAINING_EXPLAINER.md`. These were **not reproduced in this audit** — re-running
  `python -m src.detector.train` takes several minutes and would overwrite the
  existing model artifact, so it was left untouched. Treat these specific numbers
  as "documented, plausible, not independently re-verified today."

## What has NOT been validated

- **The Suricata field mapping against a real Suricata installation.** Neither
  `suricata` nor `ollama` is installed/on PATH in this environment. Every test of
  `flow_to_features()` so far uses hand-written JSON shaped like Suricata's
  documented `flow` event schema — nobody has confirmed Daniel's actual Suricata
  build emits exactly these field names with these semantics.
- **Any live integration with Daniel's router VM.** No real `eve.json` sample
  exists anywhere in this repo.
- **Any integration with Willow's dashboard/database.** There is no database
  client, no API client, no message-queue code anywhere in `src/`. The only
  "downstream" action that exists is `print()` inside `tail_eve()`.
- **The bundled demo data does not actually trigger an alert against the real
  model.** All 3 sample flows in `data/sample_eve.json` score 0, 38, and 36 — all
  below the 70 threshold. Demoing `--demo` as-is will *not* show a detected
  attack; it only shows feature mapping + a "normal" classification. (`demo.py`,
  added by this audit, fixes this — see DEMO_PLAN.md.)
- **PCAP-based testing.** There is no PCAP ingestion code anywhere in `src/`
  (confirmed via repo-wide search). The only documented/working input format is
  Suricata's live `eve.json`. "PCAP testing" as referenced in planning docs would
  require running a PCAP through Suricata's offline mode (`suricata -r file.pcap`)
  to produce an `eve.json` first — that workflow is not built or documented yet.
- **`notebooks/02_train.ipynb`.** Every cell shows `execution_count: null` and no
  cached output — it has never actually been run in this environment, even though
  it closely mirrors the logic in `train.py` (which has been run, per the model
  artifact's timestamp).
- **The Ollama report backend.** Code path exists in `report.py`/`prompts.py`, but
  `ollama` is commented out in `requirements.txt`, not installed, and the fallback
  path has never been exercised end-to-end (only the `except Exception` catch-all
  has been read, not triggered intentionally).
- **Reproducibility across machines.** `requirements.txt` only specifies lower
  bounds (`pandas>=2.0` etc.). The actual installed versions in this `.venv`
  (`pandas==3.0.3`, `numpy==2.4.6`, `scikit-learn==1.9.0`) are significantly newer
  than the floors. There's no lock file, so a teammate installing fresh today could
  get different versions than what trained the current `detector.joblib`.

## Known warnings, bugs, and risks

1. **No version control.** See above — fix this before anything else.
2. **Detector and reporter are not wired together in the live code path.**
   `suricata_reader.tail_eve()` calls `predict()` but never calls
   `generate_report()`. Each module is independently correct and tested, but
   nothing in `src/` connects them — the connecting code only existed in
   `demo.py`, which this audit added.
3. **`.gitignore` has a real gap.** It ignores `data/*.csv` and
   `data/MachineLearningCVE/`, but **not** `data/zips/MachineLearningCSV.zip`
   (235 MB) or `data/GeneratedLabelledFlows.zip` (284 MB). If a git repo is
   initialized today without fixing this, ~520 MB of redundant dataset zips would
   get committed by accident.
4. **`data/GeneratedLabelledFlows.zip` appears to be dead weight.** Nothing in
   `src/` or the notebooks references it; `train.py` only globs `*.csv` under
   `data/`. It and `data/zips/MachineLearningCSV.zip` together duplicate the
   864 MB already extracted into `data/MachineLearningCVE/` — ~520 MB that could
   be deleted with no code changes.
5. **The bundled demo data never crosses the alert threshold** (see above) —
   anyone running `--demo` for the first time, including a grader, will see only
   "normal" classifications and may conclude the detector doesn't work.
6. **No dependency pinning** — see reproducibility note above.
7. **`tail_eve()`'s only downstream action is a `print()` statement** — this is a
   known, intentional placeholder (documented in TODO.md), not a bug, but it means
   "integration with Willow" is currently 0% implemented in code, only planned.
